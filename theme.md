1. SQL vs NoSQL
SQL (PostgreSQL, MySQL)

Это реляционная БД.

Данные хранятся в виде таблиц.

Users

id | name
---------
1  | Ali
2  | John
Orders

id | user_id | price
--------------------
1  | 1       | 200

Эти таблицы связаны между собой.

Под капотом

PostgreSQL хранит таблицу не как Excel, а как множество страниц (Pages).

Каждая страница обычно 8 КБ.

Table Users

Page 1
-------
id=1
id=2
id=3

Page 2
-------
id=4
id=5
id=6

Если нужно найти пользователя с id=2 и индекса нет, PostgreSQL читает страницы одну за другой.

Disk

Page1
↓

Page2
↓

Page3
↓

...

Это называется

Sequential Scan

Плюсы SQL

✔ ACID

✔ JOIN

✔ сложные запросы

✔ надежность

✔ ограничения данных

Минусы

Если данных миллиарды,

JOIN становится дорогим.

2. Индексы

Самая любимая тема интервью.

5. Транзакции

Это группа операций.

Либо всё.

Либо ничего.

Под капотом

PostgreSQL сначала пишет изменения в журнал WAL (Write-Ahead Log), а уже потом изменяет сами страницы данных. Если питание отключится, база восстановит изменения из WAL.

BEGIN

↓

UPDATE

↓

INSERT

↓

DELETE

↓

COMMIT

Если ошибка

ROLLBACK

Все откатывается.

ACID
Atomicity

Всё или ничего.

Consistency

После транзакции база остается правильной.

Например

Баланс не станет отрицательным.

Isolation

Пользователи не мешают друг другу.

Durability

После COMMIT данные сохранятся даже при выключении компьютера благодаря WAL.

6. Уровни изоляции

Проблемы:

Dirty Read — чтение незакоммиченных данных.
Non-Repeatable Read — повторное чтение возвращает другие данные.
Phantom Read — повторный запрос возвращает новые строки.

Read Uncommitted

Самый быстрый.
Может читать грязные данные (в PostgreSQL фактически работает как Read Committed).

Read Committed

Видны только подтвержденные изменения.
Используется чаще всего.

Repeatable Read

Все данные внутри транзакции остаются одинаковыми при повторном чтении.
Подходит для отчетов.

Serializable

Максимальная защита.
Транзакции выполняются так, будто идут одна за другой.
Самый медленный, но самый безопасный.
7. Locks (Блокировки)

Когда два пользователя хотят изменить одну запись одновременно.

Под капотом PostgreSQL ставит блокировку.

User A

UPDATE balance

↓

LOCK

↓

User B

ждет
Row Lock

Блокируется только одна строка.

Самая частая.

Table Lock

Вся таблица.

Редко используется.

Shared Lock

Много пользователей читают.

Изменять нельзя.

Exclusive Lock

Только один пользователь может изменять данные.

8. Оптимизация SQL

Почему один запрос работает 5 секунд?

Потому что база делает слишком много работы.

Нужно:

использовать индексы;
выбирать только нужные столбцы (SELECT name, а не SELECT *);
писать правильные JOIN;
избегать лишних подзапросов;
фильтровать данные как можно раньше (WHERE).
9. EXPLAIN

Это способ посмотреть, что именно собирается делать PostgreSQL.

Например

SELECT *
FROM users
WHERE id=100;

Без индекса

Seq Scan

База читает всю таблицу.

После создания индекса

Index Scan

Читается только нужная часть дерева индекса.

EXPLAIN ANALYZE

Кроме плана выполнения, реально запускает запрос и показывает:

сколько строк прочитал;
сколько времени заняло выполнение;
какие операции были самыми дорогими.

Используется для поиска "узких мест" в запросах.

10. Connection Pool

Когда приложение подключается к PostgreSQL, создание соединения — дорогая операция: нужно установить TCP - (Transmission Control Protocol)-соединение, выполнить аутентификацию и подготовить сессию.



Без пула:

User1 → connect → disconnect

User2 → connect → disconnect

User3 → connect → disconnect

На каждое обращение создается новое соединение.



С пулом:

        Pool
   ┌────────────┐
   │ Conn1      │
   │ Conn2      │
   │ Conn3      │
   └────────────┘
      ↑   ↑   ↑
    User1 User2 User3

Приложение берет готовое соединение из пула, использует его и возвращает обратно.

Плюсы
быстрее работа приложения;
меньше нагрузка на PostgreSQL;
ограничение максимального числа подключений;
повторное использование соединений.
Минусы
если пул слишком маленький, пользователи будут ждать свободное соединение;
если слишком большой, можно перегрузить базу данных.

В FastAPI + SQLAlchemy пул создается автоматически через create_engine() или create_async_engine(), поэтому обычно не нужно вручную создавать новое соединение для каждого запроса.
